# RLM Template Reference (v2.7)

This guide explains when and how to use each template in `RLM/templates/`.

---

## Core Templates

### PRD-TEMPLATE.md (v1)

**Purpose**: Standard Product Requirements Document template with 13 sections.

**When to Use**:
- Generated automatically by `/discover`
- Use manually when writing a PRD from scratch
- Reference when ensuring PRD completeness

**Key Sections**:

| Section | Purpose | Critical For |
|---------|---------|--------------|
| 1. Executive Summary | Quick overview | Stakeholder alignment |
| 2. Problem Statement | What we're solving | Validation |
| 3. User Personas | Who we're building for | UX decisions |
| 4. User Stories | What users need to do | Requirements |
| 5. Functional Requirements | Feature list | Scope |
| 6. Non-Functional Requirements | Quality attributes | Architecture |
| 7. Technical Constraints | Limitations | Tech decisions |
| 8. Success Metrics | How to measure success | Validation |
| 9. Scope Definition | MVP vs future | Prioritization |
| 10. Risk Assessment | What could go wrong | Planning |
| 11. Timeline | When | Planning |
| 12. Open Questions | Unknowns | Clarification |
| 13. Appendices | Reference material | Documentation |

**Tips**:
- Fill Section 3 (Personas) carefully - they guide user stories
- Be specific in Section 5 (Functional Requirements) - vague requirements = vague implementation
- Section 9 (Scope) is critical - clearly separate MVP from future

### CONSTITUTION-TEMPLATE.md

**Purpose**: Project standards, coding conventions, and quality requirements.

**When to Use**:
- Generated automatically after PRD
- Reference during code review
- Update when adopting new standards

**Key Sections**:

| Section | Defines |
|---------|---------|
| Project Identity | Name, description, values |
| Technology Stack | Frontend, backend, database, infra |
| Coding Standards | Naming, organization, patterns |
| Testing Standards | Coverage requirements, patterns |
| API Standards | REST conventions, formats |
| Security Standards | Auth, data protection |
| Git Workflow | Branching, commits, PRs |
| Documentation | Code docs, READMEs |
| Error Handling | Categories, patterns |
| Performance | Targets, guidelines |
| Logging | Levels, formats |
| Deployment | Environments, config |

**Tips**:
- Complete Section 2 (Tech Stack) early - affects all other sections
- Section 4 (Testing) sets quality bar - don't set too low
- Section 7 (Git Workflow) prevents merge conflicts

### spec-template.md

**Purpose**: Detailed feature specification with user stories, technical design, and requirements.

**When to Use**:
- Generated for each feature by `/create-specs`
- Reference during implementation
- Verify during code review

**Key Sections**:

```markdown
Feature ID & Metadata
Overview
User Stories (with acceptance criteria)
Technical Design
  - API Endpoints
  - Data Model
  - Business Logic
Security Requirements
Performance Requirements
Testing Requirements
Dependencies
```

**Tips**:
- Acceptance criteria should be testable (Given/When/Then)
- API endpoints need full request/response examples
- Data models should include all validation rules

---

### task-template.md

**Purpose**: Individual implementation task with clear scope and acceptance criteria.

**When to Use**:
- Generated by `/create-tasks`
- Reference during implementation
- Update with completion info when done

**Key Sections**:

```markdown
Task ID & Metadata
  - Feature link
  - Type (implementation/testing/docs)
  - Priority
  - Estimated effort
  - Dependencies
  - Generation (v2.6+)

Description
Acceptance Criteria (checkboxes)
Technical Details
  - Files to create/modify
  - Implementation notes
Testing Requirements
Definition of Done
```

**Task Types**:

| Type | Description | Example |
|------|-------------|---------|
| implementation | Write new code | Create API endpoint |
| testing | Write tests | Write unit tests for X |
| documentation | Write docs | Document API endpoints |
| infrastructure | Setup/config | Configure CI pipeline |
| refactoring | Improve existing | Extract shared utilities |

**Tips**:
- Keep tasks to 1-4 hours
- One task = one responsibility
- Dependencies should be explicit
- Include `generation` field for checkpoint system

## Design Templates (v2.4+)

Templates for UI/UX engineering and design system specifications.

### design-system-template.md

**Purpose**: Complete design system specification including tokens, components, and patterns.

**When to Use**:
- Generated by `/cc-design system`
- Reference when building UI components
- Update when design system evolves

**Key Sections**:

| Section | Purpose |
|---------|---------|
| Design Philosophy | CREATIVE vs CONSISTENT approach |
| Design Tokens | Colors, typography, spacing, shadows, borders |
| Component Library | UI component specifications |
| Animation System | MINIMAL/MODERATE/RICH tiers |
| Accessibility | WCAG 2.1 AA/AAA compliance |
| Responsive Design | Breakpoints and mobile-first approach |

### component-spec-template.md

**Purpose**: Individual component design specification with all 8 states.

**When to Use**:
- Generated by `/cc-design component [name]`
- Reference during component implementation
- Validate during design QA

**Key Sections**:
- Component states (Default, Hover, Focus, Active, Disabled, Loading, Error, Empty)
- Props and variants
- Accessibility requirements
- Animation specifications
- Responsive behavior
- Usage examples

### feature-design-spec-template.md

**Purpose**: UI/UX specification for complete features.

**When to Use**:
- Generated by `/cc-design spec FTR-XXX`
- Extends feature spec with UI/UX details
- Reference for frontend implementation

**Key Sections**:
- User flows and wireframes
- Component inventory
- Interaction patterns
- Error states and feedback
- Accessibility considerations

### design-tokens-template.md

**Purpose**: Framework-agnostic design tokens with exports for multiple CSS frameworks.

**When to Use**:
- Generated by `/cc-design tokens`
- Import into projects for consistent styling
- Reference for theming and customization

**Supported Frameworks**:
- Tailwind CSS
- Material UI (MUI)
- Chakra UI
- Bootstrap
- Ant Design
- CSS Variables

### design-qa-checklist.md

**Purpose**: 117-point design quality assurance checklist.

**When to Use**:
- Run by `/cc-design qa`
- Before feature release
- During design review

**Categories (7)**:
- Visual Consistency (20 points)
- Accessibility (25 points)
- Component States (18 points)
- Responsive Design (18 points)
- Animation/Motion (12 points)
- Error Handling (12 points)
- Performance (12 points)

**Pass Threshold**: ≥90% (105/117 points)

### ux-research-template.md

**Purpose**: UX research documentation with cognitive psychology integration.

**When to Use**:
- Before major feature development
- When validating design decisions
- For usability testing documentation

**Key Sections**:
- Research goals and questions
- User testing methodology
- Findings and insights
- Design recommendations
- Cognitive psychology checklist (v2.7)

## Behavioral Economics Templates (v2.7)

### behavioral-economics-checklist.md

**Purpose**: Product design checklist based on behavioral economics principles.

**When to Use**:
- During feature design phase
- For pricing and conversion optimization
- To ensure ethical application of persuasion patterns

**Principles Covered**:

| Principle | Ethical Application |
|-----------|---------------------|
| Choice Architecture | Design defaults for user benefit |
| Prospect Theory | Frame messaging appropriately |
| Anchoring | Strategic value presentation |
| Social Proof | Genuine user activity only |
| Endowment Effect | Create ownership through personalization |
| Scarcity/Urgency | Real constraints only |
| Cognitive Load | Progressive disclosure |

**Dark Pattern Avoidance**:
- Fake countdown timers
- False scarcity for digital goods
- Hidden costs at checkout
- Difficult unsubscribe flows
- Confirmshaming language
- Misdirection in UI
- Roach motels (easy entry, hard exit)

## Prompt Pattern Templates (v2.7)

Located in `RLM/prompts/patterns/`, these provide reusable reasoning frameworks.

### root-cause-analysis.md

**Purpose**: Bug investigation and problem diagnosis using 5-Whys technique.

**When to Use**:
- Investigating test failures
- Debugging production issues
- Understanding complex bugs

**Applied By**: Coder, Tester agents

**Framework**:
1. Define the problem clearly
2. Ask "Why?" iteratively (5 times)
3. Identify systemic cause
4. Propose preventive solution

### decision-matrix.md

**Purpose**: Weighted technology and architecture selection.

**When to Use**:
- Choosing between frameworks
- Evaluating architecture options
- Making trade-off decisions

**Applied By**: Architect agent

**Framework**:
1. List alternatives
2. Define criteria with weights
3. Score each alternative
4. Calculate weighted scores
5. Document decision rationale

### comparative-analysis.md

**Purpose**: Structured comparison of alternatives.

**When to Use**:
- Feature prioritization
- Competitive analysis
- Approach evaluation

**Applied By**: Architect, Research agents

**Framework**:
1. Establish comparison dimensions
2. Gather evidence for each option
3. Create comparison matrix
4. Synthesize recommendations

### problem-decomposition.md

**Purpose**: Breaking complex tasks into manageable pieces.

**When to Use**:
- Large tasks (>4 hours)
- Multi-component features
- Unclear requirements

**Applied By**: Coder agent

**Framework**:
1. Identify core problem
2. Break into sub-problems
3. Identify dependencies
4. Sequence for implementation
5. Define success criteria

## GitHub Copilot Templates (v2.3)

Located in `RLM/templates/copilot/`, these templates enable GitHub Copilot integration in generated projects.

### copilot-instructions.md.template

**Purpose**: Repository-wide instructions for GitHub Copilot.

**When to Use**:
- Copied to `.github/copilot-instructions.md` when generating projects
- Provides RLM methodology guidance to Copilot

**Template Variables**:
- `{{PROJECT_NAME}}` - From PRD
- `{{PROJECT_DESCRIPTION}}` - From PRD
- `{{TECH_STACK_SUMMARY}}` - From constitution

### AGENTS.md.template

**Purpose**: Instructions for GitHub Copilot coding agent.

**When to Use**:
- Copied to project root as `AGENTS.md`
- Guides Copilot coding agent behavior

**Key Sections**:
- Before You Start checklist
- TDD Workflow requirements
- Task Completion checklist
- Key Directories reference

### Custom Agents (`agents/*.agent.md`)

**Purpose**: Specialized agent profiles for Copilot custom agents.

| Agent | Purpose |
|-------|---------|
| `rlm-coder.agent.md` | TDD implementation |
| `rlm-tester.agent.md` | Test writing and coverage |
| `rlm-reviewer.agent.md` | Code review and security |
| `rlm-architect.agent.md` | Architecture decisions |
| `rlm-research.agent.md` | Research and documentation |

**When to Use**:
- Copied to `.github/agents/` in generated projects
- Define specialized agent behaviors

### Prompt Files (`prompts/*.prompt.md`)

**Purpose**: Reusable prompts for Copilot Chat.

| Prompt | Purpose |
|--------|---------|
| `rlm-discover.prompt.md` | Discovery → PRD |
| `rlm-create-specs.prompt.md` | PRD → Feature specs |
| `rlm-create-tasks.prompt.md` | Specs → Tasks |
| `rlm-implement.prompt.md` | Implement single task |
| `rlm-implement-all.prompt.md` | Implement all tasks |
| `rlm-test.prompt.md` | Run tests & coverage |
| `rlm-review.prompt.md` | Code review |
| `rlm-fix-bug.prompt.md` | Bug investigation |
| `rlm-prime-feature.prompt.md` | Load feature context |
| `rlm-prime-task.prompt.md` | Load task context |

**When to Use**:
- Copied to `.github/prompts/` in generated projects
- Invoked via Copilot Chat: `@workspace /rlm-implement TASK-001`

### GitHub Actions Workflow

**File**: `workflows/rlm-task-to-issue.yml`

**Purpose**: Automatically creates GitHub issues from RLM task files.

**When to Use**:
- Copied to `.github/workflows/` in generated projects
- Triggers when task files are pushed to `RLM/tasks/active/`
- Creates issues that can be assigned to Copilot coding agent

**Features**:
- Parses task file format
- Creates issues with proper labels
- Supports manual trigger via workflow_dispatch
- Prevents duplicate issues

## Creating Custom Templates

To create a custom template:

1. Create file in `RLM/templates/`
2. Include clear section headers
3. Use placeholders like `[Description]`
4. Add example content where helpful
5. Include usage instructions at top

### Example Custom Template

```markdown
# [Template Name]

**Purpose**: [What this template is for]
**When to Use**: [When to use it]

---

## Section 1: [Name]

[Description of what goes here]

Example:
[Example content]

## Section 2: [Name]

[Description]

---

**Template Version**: 1.0
**Last Updated**: [Date]
```

---

## Template Best Practices

1. **Be Specific**: Vague templates produce vague documents
2. **Include Examples**: Show what good looks like
3. **Use Checkboxes**: For items that need verification
4. **Add Context**: Explain WHY sections exist
5. **Keep Updated**: Templates should evolve with your process
